JNINJA: Testing and Debugging

  • Errors, Exceptions and Warnings.
    1. Errors:
      1. Systems error - error in system or external program
      2. Programmer error - error cased by incorrect syntax or program logic or type.
      3. User error - error introduced by user inteeraction by entering wrong or incorrect input that the program was not built to handle.
    2. Exceptions: Events outside the scope of how a programme is expected to run the output of the program then lets us know that something not expected has occured.
    3. Stack trace: sequence of function calls leading back to when an excpetion started happening.
    4. Warnings: these are thrown by the program while running but it does not cause a the program execution to stop. Warnings need to be taken care of as they could lead to program to continue to run but produce wrong ouputs.
    5. Strict mode: ensure all warnings and exceptions are amplified, picked up and fixed while the program is in testing before production. This is recommended. To use strict mode add the strict call to the header or top of a JavaScript code. Like so: 'use strict'; Or within a function in which case the strict mode is applied only within the function. Like so: function strictly() {'use strict';};. This technique can be prefered expectially when sharing code as it ensures only the codes that you have written is using strict.
    6. Linting tools are encoureaged but they are no gurantee that your code is error proof. Linting helps to ensure uniformity and equal standards for program syntax and is good for teams to use to make program adhere to same standard and avoid sloppy code. Commonn JavaScript linting plugins are JS Lint and ES Lint
    7. Feature detection: is the way of finding out if a feature is supported in the user client before it can be used in your program. This can be done with if statements, by using window.navigator.userAgent call to find out what browser a user is running But this methods are all mostly not recommended. as it makes code more diffult to maintain.
  • Debugging in the browser
    There are methods that allow the halting of program execution at several or at any given point during execution so a programmer can check through the values of variables and thus be able to pick up an error or a failure. Breakpoints are used to stop the program excution at points in the programe. Debugging methods include the following:
    1. Trusty alert:
      The alert(); outputs the value of a variable of interest at that point during program excution so the programmer can check that the end result is correct in terms of what is expected. Use of alerts for debugging is no longer recommended.
    2. The console
      Most browser support a console eventhough thwy are not part of ECMAScript. It works similar to the alert() method but doe not stop a program execution. Some Console debugging calls:
      • console.log(); // output values of supplied variable parameters to console.
      • console.count(); // returns the number of times a function or variable was passed or called.
      • console.dir(); //displays an interactive list of objects properties.
        example:console.dire(node);// displays a DOM node list
        console.dirxml(); //display node list with decendants.
      • console.table();//displays arrays or objects in tabular format
    3. Most modern browser (including chrome and firefox have various debusgging tools besides the console and that allow adding breakpoints.
  • Error Objects: These are used to indentify specific types of error during program execution or they could be created using a constructor so the programmer can debug his code. These are:
    1. EvalError// error in the eval(); fucntion. This is deprecated.
    2. RangeError// error when a number is outside allowable range.
    3. SyntaxError//error in code syntax or typo
    4. ReferenceError// error when calling a function not defined or call to item that does not exist.
    5. TypeError//error in type of variable example string in a number field.
    6. URIError//error in universal resource indentifier or in decodeing it.
    7. InternalError//non standard error in JavaScript
    8. Error object Properties: four standard one are as follows:
      • message// returns the description of the error object
      • name//returns the name of the error constructor function.
      • stack//returns the stack trace of the error. Not recommended in production sites.

  • Test Frameworks:
    Jest is a Test Driven (TDD) created by Facebook. Using test framework is recommended. The process involves writing tests (that can fail), write code to pass the tests, refactor code, test refactored code and write more tests for new features and repeat.
